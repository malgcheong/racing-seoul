# 추억 여정 레이싱 (Memory Racing) — MVP

업로드한 사진들로 **개인화된 레이싱 맵**을 자동 생성하고, 맵 곳곳에 배치된 사진 액자·홀로그램을 지나며 추억을 회상하는 레이싱 게임.
`추억 여정 레이싱 게임_기능명세서_2026-07-07.xlsx` 기반 MVP 구현.

## 실행

```bash
npm install
npm run dev   # http://localhost:5173
```

- 사진이 없어도 **"샘플 사진으로 체험하기"** 버튼으로 바로 플레이 가능
- 개발/검증용 URL 파라미터: `?demo=1`(샘플 사진 자동 로드) `&auto=1`(즉시 맵 생성) `&autodrive=1`(자동 주행)

## 조작

`↑↓←→` / `WASD` 운전, `Shift` 드리프트(조향 강화). 3랩 완주 시 결과 화면.

## 아키텍처 (MVP)

**Three.js + Vite, 100% 클라이언트 사이드.** 백엔드 없음 — 사진은 브라우저 메모리(FileReader dataURL)에만 존재하고 어디로도 전송되지 않는다. 명세 5번(보안)의 "사진 유출 방지"를 MVP에서는 *아예 서버로 안 보내는 것*으로 충족.

```
src/
├── main.js                  # 화면 전환·업로드 UI·생성 플로우 오케스트레이션
├── analysis/photoAnalyzer.js # 사진 색상/밝기 분석 → 맵 팔레트 (캔버스 기반)
├── map/
│   ├── trackGenerator.js    # 시드 난수 + Catmull-Rom 폐곡선 트랙 생성
│   ├── decorations.js       # 지면·나무·건물·원경 산 (팔레트 색 반영)
│   └── photoObjects.js      # 사진 액자·홀로그램·수집 아이템 배치
├── game/
│   ├── game.js              # 게임 루프·랩 판정·근접 팝업·아이템 획득
│   ├── car.js               # 아케이드 차량 물리 + 로우폴리 모델
│   └── sounds.js            # WebAudio 신디사이저 효과음 (외부 에셋 없음)
└── ui/demoPhotos.js         # 샘플 사진 생성기 (캔버스)
```

## 명세 커버리지

| 명세 | MVP 상태 | 구현 방식 |
|---|---|---|
| 1.1 사진 분석 | ✅ (부분) | 지배 색상·평균 밝기·채도 추출. **객체 인식(1.1.2)·EXIF(1.1.4)는 Phase 2** |
| 1.2 동적 맵 생성 | ✅ | 시드 난수 트랙 곡선, 사진 팔레트 기반 하늘/지면/구조물/노을 무드, 매번 변형(1.2.4) |
| 1.3 생성 진행 모니터링 | ✅ | 프로그레스 바 + 단계 메시지 ("분석 중→지형→배경→오브젝트") |
| 2.1 일괄 업로드 | ✅ | 드래그앤드롭 + 파일 선택, 최대 50장, JPG/PNG/WEBP, 10MB 제한 |
| 2.2 갤러리·선택 관리 | ✅ | 그리드, 개별/전체 선택, 개수 표시, 최소 장수 시 버튼 활성화 (MVP는 5장, 명세는 10장 — `MIN_PHOTOS` 상수) |
| 2.3 삭제·분류 | ✅ (부분) | 개별/일괄 삭제 + 확인 다이얼로그. **태그 분류(2.3.4~5)는 Phase 2** |
| 2.4 약관 동의 | ✅ | 이용약관·개인정보·저작권 3종 체크 필수. **동의 내역 서버 저장(2.4.4)은 Phase 2** |
| 3.1 사진 오브젝트 배치 | ✅ | 액자(트랙 옆 균등 배치, 최소 5개 보장) + 공중 홀로그램(회전·부유) |
| 3.2 추억 회상 팝업 | ✅ | 근접 감지 → 사진+메모 팝업 + 감성 차임벨. 업로드 시 사진별 메모 입력 지원(3.2.4) |
| 3.3 상징물 아이템 | ✅ (간이) | 트랙 위 수집 오브(점수 +10/부스터 +30). **AI 상징물 인식→3D 모델(3.3.1~2)은 Phase 2** |
| 4 멀티플레이 | ❌ Phase 2 | 백엔드 필요 (아래 로드맵) |
| 5 보안 | ✅ (MVP식) | 사진이 브라우저를 벗어나지 않음. 서버 저장 도입 시 5.1~5.3 본격 구현 |

## Phase 2 로드맵 — Java(Spring Boot) 백엔드

MVP에서 백엔드를 뺀 이유: 사진 분석·맵 생성·주행이 전부 클라이언트에서 가능해 서버가 있으면 오히려 전송/저장/보안 비용만 든다. 백엔드가 **필요해지는 시점**은 아래 기능부터이며, 이때 Java가 적합하다.

1. **계정·사진 보관** — Spring Boot + PostgreSQL + S3(또는 MinIO), AES-256 저장 암호화(5.1), 동의 내역 저장(2.4.4), 데이터 삭제 API(5.3)
2. **멀티플레이(4)** — Spring WebSocket(STOMP)으로 세션 초대·위치 브로드캐스트(0.1s 틱)·텍스트 채팅. 2인 규모면 충분. ※대규모 실시간이 필요해지면 Node/Go 게임서버 분리 검토
3. **AI 분석 고도화(1.1.2, 3.3)** — 객체 인식은 Java보다 Python(FastAPI + 비전 모델) 마이크로서비스가 효율적. Spring이 오케스트레이션, FastAPI가 분석 담당 (mido/reddit-translator와 같은 구조)
4. **맵 저장·공유** — 시드+사진 세트 저장으로 같은 맵 재플레이/공유 링크

## 알려진 한계 (MVP)

- 모바일 터치 조작 미지원 (키보드 전용)
- 트랙 고저차 없음 (평면 트랙)
- 사진 "객체" 인식 없음 — 색감/분위기만 반영
- 역주행 랩 카운트는 간이 처리
